**Phase 5 — Wizard Filling, Conditional Sections & Resume** (started 2026-06-12)

Scope (PHASES.md §Phase 5): staff form list (published versions only, "continuar
preenchimento" when an in_progress response exists); flat single-page render for
unsectioned forms (default section only) and a one-section-per-page wizard for
sectioned forms (progress over VISIBLE sections, back/next, per-section client
validation, conditional sections appear/disappear live as controlling answers
change, with a warn-and-clear when a hide orphans answers); resume (answers
upserted on every navigation + `last_section_id`, "salvar e sair", reopen lands
on last section with answers intact); review screen (semantic `<h2>`/`<section>`/
`fieldset`) → `submit_response` RPC (server is the authority; one server-rejection
E2E path) → confirmation; "minhas respostas" history (submitted + in_progress).

Lead notes carried into this phase (from the Phase-1 schema + this session's read):
- **The backend authority already exists** (M4/M5): `submit_response` RPC
  (visibility eval from saved answers, required-answer check, stray-answer
  cleanup, atomic flip; sign-off check **feature-flagged OFF until Phase 6** per
  ADR 0004 — Phase 5 treats `requires_signoff` sections as ordinary sections),
  `app.eval_condition` (SQL) ↔ `evalCondition` (`src/lib/queries/conditions.ts`,
  TS mirror; **drift is phase-blocking** — keep the shared vector file green),
  `app.answer_map`. RLS: the creator alone may read/edit an `in_progress`
  response + its answers; one draft per (version, user) via
  `responses_one_draft_per_user_idx`; submitted responses/answers immutable
  (triggers). The display-item answer-rejection trigger backstops the wizard.
- **New DB work expected (thin):** an atomic **answer-save path** — upsert a
  section's answers + set `last_section_id` + bump `updated_at` in one statement,
  plus an explicit clear for orphaned answers when a controlling answer hides an
  already-answered section (the warn-and-clear). Backend chooses RPC vs.
  RLS-direct-upsert in the plan (lead steer: a `save_section_answers` RPC keeps
  it atomic + server-side and avoids N round-trips). A get-or-resume entry point
  must respect the one-draft unique index (return existing in_progress or create).
  `submit_response` is reused verbatim; the data layer maps P0010/P0011/P0012 →
  pt-BR. NO new RLS policies are anticipated — verify the audit holds.
- **Rule 9:** all reads/writes go through `src/lib/queries/responses.ts` +
  `src/lib/responses/actions.ts` (new, backend-owned); reuse the canonical
  "answerable questions" + "submitted responses" helpers — the wizard must never
  inline supabase-js.
- **Frontend:** the staff filling list is the long-deferred **"Formulários"** nav
  item (Phase 3/4 left it "em breve"); this is a NEW route group under
  `c/[slug]/` (plan-gated). Reuse the Phase-4 sanitizing Markdown renderer
  (Rule 7) for `section_text` and `read-only-tree.tsx` shape for the review
  screen. `question_explanation` → helper text via `aria-describedby`. Live
  show/skip uses the TS `evalCondition` mirror ONLY (never re-implement).
- **Sign-off:** seeded sign-off sections render as ordinary sections this phase;
  the sign-off STEP is Phase 6. Do not add sign-off UI now.

Phase 5 frontend notes:
- **Design**: applied the established "calm clinical precision" system (`globals.css` tokens, Fraunces/Spline Sans, `.animate-rise-in`/`.animate-fade-in`, motion tokens). Tasteful micro-motion: staggered card reveals, a GSAP-tweened progress bar, step cross-fades — all `prefers-reduced-motion`-guarded, dynamic GSAP import off the critical path. (The runtime `frontend-design` SKILL.md is not present in this environment, as prior phases noted; the codified design system is the source of truth.)
- **Wizard architecture**: `useWizard` is the single navigation-state source. Answers held as per-item records `{itemId,questionKey,value}` (lead steer #1) → derived `AnswerMap` feeds `evalCondition` (the TS mirror, `src/lib/queries/conditions.ts`) — the ONLY show/skip authority, never re-implemented. Live visible-step list = `useMemo` over `sections.filter(evalCondition)`; flat-vs-sectioned mirrors `read-only-tree`'s default-section rule. Orphan detection exposed as `previewAnswerChange`+`detectOrphans`+`commitAnswerChange` (lead steer #2) so F4's warn-and-clear had a clean hook.
- **Client/server boundary (Rule 9 + build correctness)**: client wizard modules import forms/responses types as **type-only**; `INPUT_ITEM_TYPES` is declared locally in `use-wizard.ts` (mirrors the Phase-4 builder fix) so the client never value-imports the server-only query module and drags `next/headers` into the bundle (this WAS caught by the build and fixed). Server-only `prepare.ts` (`server-only`) adapts `ResponseForFill`→`WizardData` + resolves signed image URLs. `wizard-runner.tsx` binds B3's server actions into the client `WizardActions` shape.
- **Two real bugs found + fixed during live verification** (not just rendering): (1) the **flat form submitted without persisting** its section → server P0011 every time; fixed by routing flat "Revisar" through `handleNext` (validate→`saveSection`→review). (2) the **submitted-response redirect raced the client confirmation**: `submitResponse`'s `revalidatePath` re-ran the wizard page, which `redirect()`-ed submitted responses to `/respostas`, clobbering the confirmation; fixed by rendering the confirmation server-side for submitted responses instead of redirecting.
- **Live click-through (local Docker, real seeded logins)** — all green, zero console errors: F1 list + Continuar/Preencher; flat form (ccih) all 4 input types render + accessible; sectioned form (farmacia) progress over visible sections + conditional valuemax 5→6→5; orphan warn-and-clear dialog → DB answers cleared; review screen (semantic sections/answers, "Sem resposta" for blanks); submit→confirmation; server-rejection (P0011) surfaced in pt-BR; "minhas respostas" history. **Local DB reset after testing → seed pristine for the tester.**
- **Routing note**: "Minhas respostas" lives at `/c/[slug]/respostas` (a sibling of `forms`, NOT under it) so the nav's prefix-match active state can't light up both "Formulários" and "Minhas respostas" on the history page.

Submitted-response "Ver" currently links to the wizard route, which shows the confirmation; the full read-only version-faithful submission viewer is **Phase 7** (will reuse `read-only-tree.tsx`).
